Lesson 2: The Magic of Middleware

Now that you know the server is just a program waiting for requests, let's talk about how it actually handles them. In Express, this is all about Middleware.

What is Middleware? Imagine a request is a person walking through a series of security checkpoints before reaching the main office. Each checkpoint (middleware) can:
  • Inspect the person (Read the request)
  • Change something about them (Modify the request)
  • Kick them out (Send a response immediately)
  • Or just wave them through to the next checkpoint (Call next())

1. The Pipeline Pattern

In your index.ts, you see a lot of app.use() calls. Every time you use app.use(), you are adding a new checkpoint to the pipeline.

// The Request Pipeline in your code:
1. CORS Middleware (Check: Is this React app allowed to talk to me?)
2. Rate Limiter (Check: Is this IP sending too many requests?)
3. JSON Parser (Action: Convert the raw text body into a JS object)
4. Route Handlers (The destination: /api/v1/search, etc.)

2. The Order Matters

Because middleware is a pipeline, order is everything. If you put the JSON parser after your route handler, the route handler won't be able to read the request body because it hasn't been parsed yet!

Real-world observation: Look at your code. The limiter is applied to "/api". This means any request to /status skips the rate limit entirely, but any request to /api/v1/search must pass through the limiter first.

3. `app.get` vs `app.use`

  • app.use(): General purpose. It usually applies to a range of URLs (or all of them) and is typically used for setup/security/parsing.
  • app.get(): Specific. It only triggers when the exact URL and the GET method match.
