Lesson 1: From Browser to Server

Since you know React and Next.js, you're already halfway there. But there's a fundamental shift in how code executes when you move from the frontend to the backend.

The Core Difference: In React, your code runs in the user's browser (Client). In Node.js, your code runs on a computer (Server) that stays on 24/7, waiting for someone to send it a request.

What is Node.js?

Node.js is not a language; it's a runtime. It takes the V8 engine (the thing that powers Chrome) and lets it run on your machine instead of inside a browser. This means you can use JS/TS to interact with the operating system: read files, talk to databases, and listen to network ports.

Enter Express.js

Node.js by itself is quite low-level. Express is a framework built on top of Node.js that simplifies the process of creating a web server. If Node.js is the engine, Express is the dashboard and steering wheel.

In your index.ts, you see: const app = express();. This creates an Express application object that handles the routing of incoming requests.

The Request-Response Cycle

Everything in Express revolves around this loop:

  1. Request (req): A client (like a React app) sends a request to a URL (e.g., GET /status).
  2. Processing: Express looks at the URL and runs the corresponding function.
  3. Response (res): The server sends back data (usually JSON) and a status code (e.g., 200 OK).
Your First Win: You can now look at app.get("/status", ...) in your code and realize it's just a function that waits for a GET request to /status and sends back a JSON response.
